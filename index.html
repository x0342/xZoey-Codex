第一部分：AI 大模型地图

大模型不只是聊天机器人，Codex 属于工程 Agent

我们先用一张地图建立直觉：模型提供理解和生成能力，产品把能力包装成入口，Agent 则在模型外面加上工具、计划、权限和反馈循环。Codex 站在这个 Agent 层，面向真实软件工程任务。

  • 1
    语言模型
    理解文字、生成文字、解释概念、总结材料。普通聊天 AI 主要让人感觉到这类能力。
  • 2
    多模态模型
    可以同时处理文字、图片、声音、视频等输入。它让 AI 不只读文字，也能看界面、看截图和听内容。
  • 3
    工具型 Agent
    模型不只是回答，而是能读文件、调用命令、打开网页、查资料、做步骤规划，逐步完成任务。
  • 4
    工程 Agent
    把 Agent 放进代码仓库里，它就要围绕需求、文件、测试、diff、review 和交付来工作。Codex 就是这一类。

2021：补全

Copilot 时代，AI 像更聪明的输入法。开发者写开头，它补后面，但项目拆解、测试和交付仍主要靠人。

2022：对话

ChatGPT 让 AI 从补全进入问答。它能解释报错、写片段、给建议，但通常不在真实项目现场。

2023-2024：协作

Cursor 这类 AI 编辑器让 AI 进入编辑器，能看文件、改局部、跨文件协作，开始贴近项目上下文。

2025：工程 Agent

Codex 的重点从“辅助写代码”转向“推进工程任务”：读项目、改代码、跑命令、检查结果。

第二部分：从模型到 Agent

关键转折：AI 从“给答案”变成“推进任务”

普通聊天工具主要给建议；AI 编辑器贴近项目修改；Codex 更进一步，把需求拆成步骤，在仓库里读文件、改文件、跑命令、看 diff，并把结果交给人审查。

ChatGPT
顾问式回答

适合解释概念、分析思路、生成片段。问题是上下文、代码迁移和验证仍要由人完成。

Cursor
编辑器协作

AI 进入编辑器，能围绕当前项目做局部修改。它更贴近代码，但通常仍需要开发者持续盯住下一步。

Codex
工程执行

面向软件开发任务，围绕仓库、命令、测试、diff、review、PR 和交付结果来工作。

第三部分：Codex 能力拆解

Codex 的核心能力：先理解项目，再交付可审查结果

这里要讲清一个关键区别：Codex 不是只会生成代码片段，而是能沿着工程任务往前走。它先读懂陌生项目，解释代码逻辑，再处理 bug、功能、测试、重构、文档和 PR，最后把命令结果与 diff 留给人审查。

Review Summary

第四部分：Codex 的入口和场景

同一个 Codex，可以在 App、CLI、IDE 和 Web 之间流转

刚上手优先从 Codex App 开始，因为图形界面能看见项目、线程、计划、命令和 diff。熟悉 Git 与终端后，再把 CLI、IDE Extension、Web / Cloud 接进日常开发和团队协作。

Codex App

项目目录就是 Codex 进入的工作现场；thread 是一个具体任务；review pane 用来检查改了哪些文件和行。

  • 一项任务一个 thread
  • 看绿色新增和红色删除
  • 确认后再接受改动

Codex CLI

终端里的本地仓库工作方式：进入项目目录，启动 Codex，复杂任务先 `/plan`，修改后用 `/diff` 看结果。

  • 读写本地项目
  • 运行测试和构建
  • 适合 Git 工作流

IDE Extension

在 VS Code、Cursor 或 Windsurf 旁边使用 Codex，适合围绕当前文件、选中代码、局部 bug 和小功能协作。

  • 贴近代码编辑
  • 适合局部修改
  • 适合日常开发

Web / Cloud

连接 GitHub 仓库后，可以让 Codex 在云端处理任务、开分支、整理变更并进入 PR 审查流程。

  • 后台处理任务
  • 适合多人协作
  • 适合 PR 流程

Browser Use

前端任务需要看真实页面：打开网页、验证交互、截图检查、确认移动端布局，这些步骤把代码改动连接到结果。

  • 验证前端页面
  • 检查响应式
  • 辅助调试 UI

Computer Use

当任务跨出编辑器，Codex 也可以借助桌面操作能力处理应用界面。能力越广，权限边界越要清楚。

  • 桌面操作
  • 跨应用流程
  • 需要更严格边界

第五部分：最重要的使用方法

稳定使用 Codex，不靠神奇提示词，靠工作流

稳定使用 Codex 的核心不是背神奇提示词，而是把任务放进可控流程：先说清目标和边界，再建立上下文、制定计划、小步实现、运行验证、检查 diff，最后人工验收。

01

需求拆解

不要只说“帮我优化”。要说目标、范围、限制和验收标准。Codex 的执行质量，很大一部分取决于边界是否清楚。

关键理解：人负责定义方向和验收标准，Codex 负责推进过程。目标不清楚时，它推进得越快，偏得越远。
只优化首页首屏，不改登录和支付逻辑。
02

建立上下文

让 Codex 先读 README、目录结构、配置、入口文件、测试命令和 AGENTS.md。它需要理解项目现场，而不是凭印象写代码。

先读项目：一个新人接手项目也不会马上改代码，Codex 也一样要先知道文件在哪里、入口在哪里、哪些地方不能碰。
$ rg --files
$ cat package.json
03

计划制定

复杂任务先让 Codex 列计划。计划不是形式主义，它能把风险、顺序、文件范围和待确认点提前暴露出来。

可讨论：计划把 AI 的内部推进行为变成外部对象；人可以确认、修改，或让它缩小范围。
读文件 → 改结构 → 改样式 → 截图验证
04

小步实现

一次只改一个目标，避免“顺手重构”。小步意味着出错时容易定位，diff 也容易审核。

控制范围：先做窄任务，再逐步扩大范围。不要第一步就把整个系统交给 Codex。
只改视觉层，不改业务逻辑
05

测试验证

能跑测试就跑测试，能跑 build 就跑 build，前端页面还要截图看实际效果。Codex 的价值也包括协助验证。

完成标准：“AI 写了”不是完成，“AI 写了、跑过验证、能解释结果”才接近完成。
$ npm run build
✓ build passed
06

代码审查

不要只看总结。真正要检查的是实际修改了哪些文件、哪些行、是否碰了不该碰的地方。

看 diff：绿色是新增，红色是删除。review 是人机协作里最关键的一步，因为责任仍然在人。
+ video demo section
- forced pinned scroll
07

人工验收

Codex 可以执行，但最终判断仍然是人：功能是否正确，视觉是否合适，风险是否可接受。

责任边界：Codex 是执行者，不是责任转移器。无法验收的任务，就不适合让它独立完成。
确认功能、体验、风险和交付标准
08

交付复盘

满意后再提交、开 PR 或复盘。重复出现的规则可以写进 AGENTS.md，重复流程可以沉淀成 Skill。

沉淀方法：一次任务是执行，多次任务要沉淀规则、模板和技能，下一次才会更稳。
commit / PR / AGENTS.md / Skill

第七部分：安全和可控

Codex 能动手，所以必须先有边界

Codex 可以读取、修改文件，也可以在项目目录里运行命令。正确做法不是完全害怕，也不是完全放权，而是用项目目录、权限审批、Git、diff、worktree 和人工验收把风险锁住。

第八部分：进阶扩展

Plugin、Skill、MCP：安装能力、固定方法、连接工具

Codex 不是孤立聊天框。重复出现的项目规则可以写进 AGENTS.md，固定流程可以变成 Skill，外部工具和数据可以通过 MCP 连接，多个能力还可以打包成 Plugin。

AGENTS.md

项目级说明书，记录技术栈、命令、代码风格、边界、验收方式和不要触碰的区域。

Skill

固定“怎么做”。例如代码审查、生成页面、写文档、处理设计稿，都可以沉淀成可复用流程。

Plugin

解决“怎么安装和分发能力”。它可以把 Skill、MCP、资源和配置打包成一个可安装能力包。

MCP

解决“连接什么工具和数据”。数据库、浏览器、GitHub、文档系统，都可以通过接口接入。

Hooks

在关键生命周期前后做强制检查，例如命令前确认、提交前验证、危险操作前暂停。

Automations

让 Codex 定期检查、提醒、复盘或继续处理任务，适合重复巡检和长期项目维护。

最后：把路线串起来

从上手到实战，Codex 的学习路线可以这样走

最后把整条路线收束成八步：先知道资料怎么读，再理解 Codex 的定位，准备账号、系统和项目目录；随后进入核心功能、标准工作流、任务模板和实战案例。听完这一页，观众应当能判断自己下一步从哪里开始。

0. 使用说明

先交代资料性质、适合人群和阅读路线：快速上手、开发者核心路线、进阶扩展路线。

1. 先搞懂 Codex

讲 Codex 基础认知、AI 编程四阶段、能做什么，以及和 ChatGPT、Cursor、Claude Code 的定位差异。

2. 安装与准备

按账号、系统、工具、项目目录、权限安全来准备；先用练习目录，不直接上重要项目。

入口怎么选

App 适合新手和演示；CLI 适合终端和 Git；IDE 适合局部开发；Web / Cloud 适合 GitHub 和团队任务。

3. 核心功能详解

继续展开自动化、插件、Skill、MCP、代码管理、云端运行、记忆系统这些工程化能力。

4. 标准工作流

需求拆解、制定计划、小步实现、测试、代码审查、提交与复盘；每一步都有失败原因和检查点。

任务模板库

把读项目、修 bug、加功能、写测试、做 review、写 PR 这些场景做成可复用提示模板。

5. 实战案例库

从前端页面、功能优化、管理后台，到 PPT 和宣传视频，让 Codex 从代码扩展到交付物。

最终理解：橙皮书的核心路线是：先理解 Codex，再准备环境，再用标准工作流交付低风险任务，最后把重复方法沉淀成模板、规则和 Skill。
